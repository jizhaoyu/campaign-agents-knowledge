# 审批流设计

## 目标

避免 Agent 直接执行高风险业务动作，确保关键动作进入人工确认。

## 第一版审批对象

- 创建正式工单

后续可扩展：

- 自动分派工单
- 自动关闭工单
- 自动通知外部系统

## 状态流转

### 工单状态

`DRAFT -> PENDING_APPROVAL -> OPEN`

或

`DRAFT -> PENDING_APPROVAL -> REJECTED`

### 审批任务状态

`PENDING -> APPROVED`

或

`PENDING -> REJECTED`

## 触发规则

满足以下任一条件需要审批：

- 工单优先级为 HIGH
- 建单内容由 Agent 自动生成比例较高
- 提交者角色不具备直接开单权限

## 审批步骤

1. 用户确认工单草稿
2. 系统判断是否需要审批
3. 若需要，则创建审批任务
4. 审批人查看草稿、引用证据和上下文
5. 审批通过则创建正式工单
6. 审批驳回则保留草稿并记录原因

## 审批页面需要展示的信息

- 申请人
- 草稿标题
- 草稿描述
- 优先级
- 证据引用
- 相似历史工单
- 会话摘要

## 审批备注模板

第一版内置标准备注模板，按通过和驳回分别维护：

- 通过：证据充分，批准开单
- 通过：业务连续性优先
- 驳回：证据不足，退回补充
- 驳回：疑似重复工单

审批人可以选择模板并二次编辑最终备注。系统保存最终备注，并在审计 payload 中记录 `templateCode`，用于区分标准模板和自由输入。

## 审批审计要求

记录：

- 审批人
- 审批时间
- 审批结果
- 审批备注
- 关联 traceId
